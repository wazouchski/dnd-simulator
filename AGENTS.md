# AGENTS.md

## Project

This workspace is for the **D&D Character Balance Tester**, a browser-based D&D 5e combat simulator.

The product lets users import a D&D Beyond character, configure an SRD encounter, run client-side combat simulations, and inspect aggregate results and round-by-round logs.

## Current State

The first Next.js + TypeScript vertical slice exists.

Implemented:

1. Static-export-compatible Next.js app.
2. Domain types for characters, monsters, encounters, combat logs, and simulation results.
3. Small SRD 5.1 monster seed set.
4. Sample single-class fighter.
5. Deterministic dice utilities with optional seeded mode.
6. Basic attack-roll combat with initiative, HP tracking, hits, misses, critical hits, and damage rolls.
7. Two-step tactical workbench UI:
   - setup screen for character import, D&D Beyond bookmarklet handoff, loadout verification, manual attack entry, and encounter controls
   - simulation screen for playable turn-by-turn scenes and batch odds
   - aggregate results and expandable combat log
8. D&D Beyond parser support for equipped weapon rows, attack-roll spell rows, `classSpells` spell buckets, non-damage/control/utility spell sheet actions, fallback skills, imported sheet summaries, Public-character errors, and manual JSON fallback.
9. Character sheet review uses tabs for Actions, Spells, Inventory, Features & Traits, Background, Notes, and Extras.
10. Loadout review supports weapon equip checkboxes so imported inventory weapons can be included/excluded before simulation.
11. Setup review supports manual corrections for AC, HP, initiative, proficiency bonus, speed, base ability scores, missing attack actions, save-spell actions, and manual feats.
12. Playable mode includes first-pass save-spell handling for control/debuff spells such as Hold Person.
13. Current UI direction is **Dragonforge Oracle**, a high-fantasy tactical artifact skin: obsidian panels, gold inlay, bloodstone/emerald/violet accents, spellbook-like inputs, arcane titlebars, a d20 sigil asset, and simulator panels mapped to sanctum/hero dossier/arsenal altar/monster grimoire/quest briefing/battle altar/oracle results.
14. Electron desktop packaging:
   - desktop app serves the static `out/` export through a local-only HTTP server, usually `127.0.0.1:3217`
   - `/api/import` and `/api/import/events` bridge bookmarklet imports into the Electron window
   - Windows artifacts are built with `npm.cmd run desktop:dist` into `release/`
   - current friend-test handoff is the one-click installer at `release-current/DND-Character-Balance-Tester-Friend-Test-Setup.exe`
   - desktop icon is generated by `npm.cmd run icon` and applied by `electron/after-pack.cjs`
   - self-signed local builds use `npm.cmd run desktop:dist:selfsigned:trust`; public cert is exported to `build/certs/dnd-character-balance-tester-local.cer`
15. For reliable local browser preview on Windows, prefer `npm.cmd run build` then `npm.cmd run preview:static` on port `3001`; `next dev` can wedge in this sandbox.

## Product Rules

- Local-first: character data must stay in the browser for MVP.
- No accounts, backend storage, telemetry, or server-side D&D Beyond scraping.
- Single-class characters only for v1.0.
- SRD monsters only for v1.0.
- Do not build a VTT, campaign simulator, full party simulator, or full spell simulator for MVP.
- Be transparent about approximations. Combat logs should expose rolls and modifiers.
- Prefer the actual simulator workspace as the first screen, not a marketing landing page.
- Friend-test onboarding must assume non-technical users. Avoid requiring users to understand JSON, bookmarklets, or browser internals before they can run a sample character.

## Architecture Direction

- Use Next.js with TypeScript.
- Keep the app static-export-compatible for subdomain hosting.
- Keep the Electron wrapper pointed at the same static export so web hosting and desktop packaging share one app.
- Keep browser-only APIs out of server components.
- Keep the combat engine independent from React UI.
- Move large simulation runs to a Web Worker when needed.
- Prefer small, testable pure functions for rules math.

## Quality Bar

Before considering work done:

- Code runs without errors.
- Core behavior is manually verified.
- Non-trivial rules logic has tests.
- Accessibility basics are respected.
- State docs/logs reflect the work.
- Known issues are logged in `03-logs/bug-log.md`.

## Open Questions to Keep in Mind

From `00-context/assumptions.md`:

- How sophisticated should enemy tactics be for MVP?
- Are concentration spells ignored or approximated?
- What is the canonical SRD monster data source?
- How much variance is acceptable with the default 20 iterations?
- How should bookmarklet handoff work: URL parameter, localStorage handoff, or another browser-only flow?

Make conservative choices that keep the MVP useful and local-first.

## Current Priority

Priority 1 is reliable D&D Beyond character import and spell coverage. Battle simulation quality matters, but the product differentiator is trustworthy full character-sheet ingestion first:

- streamline bookmarklet/import onboarding in plain language for non-technical testers
- import all visible character-sheet spell data instead of only damage spells
- import and review visible weapons, armor, shields, inventory, and AC signals
- separate rollable attacks from full sheet actions
- model common control/setup spells, starting with Darkness and Hold Person
- preserve user-confirmed loadout edits across re-imports

The target UX flow is:

1. Character Sheet Upload / Import
2. Character Sheet Review / Modifications
3. Mode Select
4. Batch Results or Playable Encounter

Playable Encounter should become replayable and entertaining, with visible character details, skills, spells, movement, resources, varied encounter beats, terrain/position prompts, and class-specific choices.
